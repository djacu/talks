Every setting has a priority

Lowest number wins.

Reveal the rows top to bottom:

One module, two overrides


# nixpkgs nginx module (systemd hardening)
ProtectSystem = "strict";      # plain, so 100
ProtectHome = mkDefault true;  # a default, so 1000

Serve from /home? ProtectHome = false; just wins. (100 beats 1000)

Loosen ProtectSystem? Same priority, a conflict. Reach for mkForce.

Show the two lines, then the two outcomes:

  1. ProtectHome: "The module set this as a default. Need nginx to serve a user's files from /home? Write ProtectHome = false. You're 100, the default is 1000, lower wins, done. No ceremony."
  2. ProtectSystem: "This one the module set plainly, also 100. Your normal override ties it, and a tie is a conflict, an error. That's the module saying it meant this. So you say mkForce and outrank it. The error isn't in your way, it's a seatbelt."

Overrule the package

glow's word-wrap fix has sat unmerged for months. Take it anyway:


glow.overrideAttrs (old: {
  patches = (old.patches or [ ]) ++ [ (fetchpatch {
    url = "github.com/charmbracelet/glow/pull/886.patch";
    hash = "sha256-...";
  }) ];
})
  1. The problem: "There's an open PR on glow that fixes word wrapping. It has sat there for months. On most systems you wait, or you fork and build it by hand."
  2. overrideAttrs: "On Nix you take the package as it is and change one attribute: its patch list."
  3. fetchpatch: "You point fetchpatch straight at the pull request's .patch URL. GitHub serves any PR as a patch. Pin the hash, and your glow has the fix. You didn't wait for the maintainer. Third rung: overrule the package."

A maintainer can burn it down

January 2022: the author of colors.js shipped an infinite loop on purpose. Thousands of apps broke overnight. 


src = fetchFromGitHub {
  owner = "...";
  repo = "...";
  rev = "v1.4.0";       # you chose this version
  hash = "sha256-...";  # and pinned this hash
};

A hostile release can't reach you until you bump the hash.

  1. The sabotage: "In 2022 the maintainer of colors.js, millions of downloads a week, intentionally shipped an infinite loop. Anything that auto-pulled the latest version broke instantly."
  2. The pin: "Nix fetches sources by exact version and content hash. You chose the version; you pinned the hash."
  3. The point: "A new release, sabotaged or just broken, cannot enter your build until you change that hash yourself. Auto-update can't hurt you, because there is no auto-update."

The one thing you can't override

August 2023: HashiCorp relicensed Terraform to the BSL, overnight.

You can't patch a license.

So the community forked it. nixpkgs marked Terraform unfree and shipped OpenTofu the same day.

  1. The rug-pull: "In August 2023 HashiCorp relicensed Terraform from open source to the Business Source License, overnight. Same code you'd used for years, new terms, no say."
  2. The honest limit: "Here's the one thing the ladder can't do. You cannot override a license. No patch, no overlay, changes the terms."
  3. The exit: "But this is open source's oldest move: you don't override it, you leave, and you take the code with you. The community forked OpenTofu, and nixpkgs marked Terraform unfree and shipped OpenTofu the same day. Override when you can; exit when you can't."

The repo vanished. The build didn't.

October 2020: an RIAA takedown wiped youtube-dl off GitHub.

Nix users kept right on installing it.

Nix addresses sources by content hash and caches the result. Delete the upstream, and the exact bits are still there, still verifiable.

  1. The takedown: "October 2020. The RIAA sent GitHub a DMCA notice, and youtube-dl's repository vanished."
  2. Kept installing: "But Nix users kept installing it, the same afternoon."
  3. Why: "Nix doesn't fetch from wherever a project happens to live. It fetches a source by its content hash, and caches the built result by hash too. Delete the GitHub repo and the exact bits are still in the cache, still verifiable. A takedown removes a URL; it can't remove a hash."

The license fights the distro

Normally you'd patch it out. But its license forbids distributing a modified build. So nixpkgs removed the package: 


# nixpkgs aliases.nix
duckstation = throw
  "removed following upstream request";
  1. Why not just patch it: "Normally you'd do exactly what we did with glow, patch the check out. But DuckStation's license forbids distributing a modified build script."
  2. The removal: "So nixpkgs literally cannot ship the workaround. It did the only thing it could: removed the package. That throw is what you get now."

But it's still yours

The distro can't distribute the patch.

You can still apply one on your own machine. That isn't distribution.

The license binds the distro. It doesn't bind you.

  1. The distinction: "Is this the one case where you lose? No. The license stops the distro from shipping a patch to everyone. It does not stop you from applying one on your own machine. Running a patched build for yourself isn't distribution."
  2. Land it: "The license binds the distro. It doesn't bind you. Even the most hostile upstream can't take the source out of open source."
